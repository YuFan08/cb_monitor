## Strict Engineering Standards (顶级工程标准)

请严格遵循以下现代 Python 顶级企业级开发规范：

1. 项目管理与大一统配置：
   - 全局使用 `uv` 进行依赖与虚拟环境管理。
   - 彻底摒弃杂乱的配置文件，所有的项目元数据、依赖、工具配置（Lint/Format/TypeCheck）必须统一收拢于 `pyproject.toml`。
   - 新项目基于 Python 3.14 最新特性编写。
2. 架构模式与解耦：
   - 采用标准的 `src` 目录结构（如 `src/CB_Tools/...`）。
   - 核心业务逻辑（网络抓取、网页解析）与外部副作用逻辑（本地 I/O，FFmpeg下载）必须强解耦。预留 `tests/` 目录，确保代码具备高可测试性。
3. 极简工具链 (Lint & Format)：
   - 废弃 Black/isort/Flake8，全局统一使用 `Ruff` 处理代码格式化（Formatter）和静态分析（Linter）。
4. 极致的类型安全 (Type Safety)：
   - 静态检查：全量使用 Python 3.14 现代 Type Hints，并符合 `Pyright` (Strict 模式) 的安全要求。
   - 运行时校验：外部爬取的网络数据、解析参数等，必须使用 `Pydantic` V2 模型进行解析与拦截，落实 Fail-Fast（快速失败）原则。
5. 健壮性与异常处理：
   - 永远不信任外部输入。精准捕获特定异常（如 `httpx.HTTPError`），严禁使用裸 `except:` 或 `except Exception:`。
   - 核心网络请求必须具备完善的重试与超时机制。
6. 现代化可观测性：
   - 严禁使用 `print()` 打印业务流程。引入 `Loguru` 处理所有终端输出与文件日志，提供美观且带有上下文的日志记录。
7. Pythonic 与代码整洁高效：
   - 整个项目架构使用极致的异步架构，使用httpx配合selectolax爬取并解析，让效率达到极致。
   - 深度且合理地应用推导式、上下文管理器 (`with`)、生成器 (`yield`)。
   - 遵循单一职责原则 (SRP) 和 DRY 原则。函数长度严格控制，逻辑过深必须拆分。
   - 使用 Google 风格 Docstrings；注释只写 "Why"（业务背景/妥协原因），不写 "What"。
   - 整个项目代码必须严格遵循PEP8的规范（这点非常重要），让整个项目源码具有极强的可读性和管理方便性。
   - 生成后的代码要自动使用ruff格式化和检测，后续我提出修改要求，也是如此
8. 专注细节和效率（这点非常重要）：
   - 选择最合适的依赖
   - 避免逻辑 Bug
   - 避免冗余操作和不必要的开支消耗，用最少的代码做最多的活，让代码高效运转，
9. UI界面美观（颜值就是生产力，我是颜值控）
   - UI输出日志，语句简洁，合理，逻辑清晰，一目了然
   - 配色美观，或者使用Loguru默认配色F
10. 保证整个项目一致性和后续部署使用稳定

- 单个模块的修改要同步到其他与之相关的文件
- 后续项目会移植到Ubuntu服务器，编写代码时要注意本系统和服务器ffmpeg版本的差异
